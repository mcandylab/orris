# Upgrade Fastify v4 → v5

**Date:** 2026-03-12 14:56
**Files:** backend/package.json, backend/src/index.ts
**Severity:** high

## Problem

Fastify v4.x had two high-severity DoS vulnerabilities:
- GHSA-mrq3-vjjr-p77c: Unbounded Memory Allocation in sendWebStream
- GHSA-jx2c-rxcm-jvmq: Content-Type tab character allows body validation bypass

Additionally, v4 is no longer actively maintained for security patches.

## Root Cause

Project was initialized with `fastify: "^4.28.1"`. Fastify v5 was released and all
ecosystem plugins (`@fastify/cors`, `@fastify/jwt`, `@fastify/rate-limit`, `fastify-plugin`)
released major versions requiring Fastify v5 as peer dependency.

## Solution

Updated all Fastify-related dependencies to their v5-compatible versions:

```json
"fastify": "^5.8.2"           (was ^4.28.1)
"@fastify/cors": "^11.2.0"    (was ^9.0.1)
"@fastify/jwt": "^10.0.0"     (was ^8.0.1)
"@fastify/rate-limit": "^10.3.0" (was ^9.0.0)
"fastify-plugin": "^5.1.0"   (was ^4.5.1)
```

**Breaking change in Fastify v5:** `setErrorHandler` now types the error parameter
as `unknown` by default (stricter TypeScript). Fixed by explicitly typing the generic:

```typescript
// Before (v4 — error was implicitly FastifyError)
app.setErrorHandler((error, request, reply) => { ... })

// After (v5 — explicit type annotation required)
app.setErrorHandler<FastifyError>((error, _request, reply) => { ... })
```

Also added `FastifyError` import from `fastify`.

## Prevention

- When initializing new projects, pin to the latest major of Fastify (v5+)
- Run `npm audit` before each commit checkpoint to catch new vulnerabilities early
- All `@fastify/*` plugins have their own major versions aligned with Fastify's major version —
  upgrade them together (e.g., Fastify v5 → @fastify/* v10+)
